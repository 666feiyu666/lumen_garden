# 01 种植花谱原型开发 Harness Prompt

以下提示词用于将《萤光花园》的下一阶段开发交给编码 agent。它固定本次设计评审后的方向：经典图案应成为灯灵种植后由纯规则验证的谜题目标，而不是独立的被动观察模式。

执行记录：`01` 已于 2026-05-26 完成最小纵向切片，交付 `Block / 四叶眠床` 种植题、三种经典行为规则测试及无窗口 UI 烟雾测试；`Glider / 流光种` 可玩关卡留待后续编号任务。

```text
你正在继续开发 Python + pygame 游戏《萤光花园 / Lumen Garden》。

任务目标
========
在保留当前“花园谜题”玩法与教程的前提下，新增“种植花谱 / Planting Bloom”的最小可玩谜题原型：

- 当前花园谜题，也可在未来导航中称为“自由演化”：玩家移动或停留灯灵；灯灵作为额外邻居影响每一轮生长；结算目标为规定步数后的盛开数量范围。
- 新增种植花谱：玩家控制灯灵在静止花圃中移动，消耗有限花种补全或构造起始布局；确认开始后，灯灵不参与邻居计算，花芽以纯 `B3/S23` 演化并验证经典行为。
- 首版必须包含实际播种决策和结果验证，不以只可翻看的观察册替代谜题。

开始前必须做的事
================
1. 阅读当前 README、docs/game-design-document.md、docs/classic-mode-design.md、docs/tutorial-level-design.md 和 docs/game-flow.md。
2. 阅读 main.py、包内 app.py、model.py、puzzles.py、tutorial.py，以及 tests。
3. 运行现有测试，确认基线是否通过；如果原有测试失败，先描述失败，不要把失败误归因于新改动。
4. 确认实际包目录大小写。本工作区现有可运行包目录为 `Lumen_Garden`，应延续该名称；不要为了整理大小写而破坏当前可运行工作区，除非该变更已被明确纳入任务并验证跨平台路径。

不可更改的设计决定
==================
- 叙事与视觉维持温暖的月光花园、萤光花朵和花谱语汇；不要恢复“变异细胞”、实验室、感染或惊悚生物意象。
- 当前教程和五个花园谜题必须继续可玩，并保持灯灵每步影响演化的现有规则。
- 种植花谱中的灯灵不是装饰：它必须用于播种。但在玩家启动生长验证后，灯灵不得作为额外邻居影响经典图案行为。
- 种植阶段移动灯灵不触发生长；首版的主要预算是有限花种与可播种位置，而不是移动步数。
- 首版不允许生长验证途中继续播种或移除花芽。
- 优先复用现有 pygame 绘制风格和布局，不引入新的 UI 框架。
- 保持修改聚焦，不做与种植花谱无关的大规模重构。

首批经典行为
============
为规则与目标判定支持三个标准图案，同时显示标准名称与花园别名：

1. `Block / 四叶眠床`
   - 类型：Still life
   - 标准验证：连续若干代保持形状一致。
2. `Blinker / 摇曳花枝`
   - 类型：Oscillator
   - 标准验证：周期为 2。
3. `Glider / 流光种`
   - 类型：Spaceship
   - 标准验证：在足够大的有限棋盘上，第 4 代为指定标准形状平移 `(1, 1)`。

首版仍使用有限棋盘且不环绕。为 `Glider` 提供足够边距，确保验证发生在其碰触边界之前。

最小可玩内容
============
首版的硬性玩家体验目标是完成至少一则种植谜题：

- 谜面应含预置花芽和少量可用花种，避免从空棋盘照图摆放完整答案。
- 玩家可移动灯灵，在合法土格播种，并撤回本次尝试中自己播下的花芽。
- 玩家确认“开始生长”后，布局锁定并以纯规则推进。
- 成功必须基于稳定、周期或迁移行为判定，而不是只判断最终花芽数量。

优先以 `Block` 或 `Blinker` 做第一则完整纵向切片；在该循环可玩并测试通过后，继续制作 `Glider / 流光种` 放飞谜题。若时间仅够一个可玩关卡，仍应为另外两种标准行为保留模型测试与数据接口，而不要用观察册冒充已完成谜题。

实施分期
========
M0：基线检查
- 确认项目结构、文档与测试状态。
- 保留当前花园谜题和教程行为。

M1：纯规则与经典行为检测
- 从当前生长逻辑中抽取可独立测试的单代演化能力，例如：
  `grow_once(blooms, width, height, extra_lights=frozenset())`
- 当前花园谜题继续在每轮传入灯灵位置。
- 种植花谱仅在开始生长后调用演化，并始终传入空光源。
- 为纯规则下的 `Block`、`Blinker`、`Glider` 标准行为添加单元测试。

M2：种植谜题数据与状态循环
- 新增 `patterns.py` 或符合现有结构的等价模块，集中定义经典名称、花园别名、起始谜面、花种预算、可播种区域和验证条件。
- 新建清晰的种植状态，或以符合现有架构的方式区分：
  - 播种阶段：移动、播种、撤回、花种余额。
  - 生长验证阶段：锁定布局、逐代演化、成功/失败、重置。
- 不让种植状态悄悄改变既有 `GameState` 的玩家体验。

M3：种植花谱 UI 纵向切片
- 在主菜单加入模式选择：“自由演化 / 花园谜题”和“种植花谱”。
- 至少提供一则完整可玩的 `Block` 或 `Blinker` 种植题。
- 种植画面至少支持：
  - 移动灯灵
  - 在合法位置播种
  - 撤回玩家本轮播种，或重置布局
  - 确认开始生长
  - 在验证阶段推进/播放所需代数并显示结果
  - `Esc` 返回
- 在说明区显示剩余花种、阶段、标准图案名称、类型和胜利验证依据。

M4：流光种招牌谜题
- 设计一则 `Glider / 流光种` 谜面，要求玩家用有限花种形成或催生能够自主移动的结构。
- 验证其在不受灯灵影响的生长阶段到达目标轮廓或通过目标区域。
- 若该谜题需要额外试玩调优，可以在完成 M3 后明确记录为下一小步，不牺牲已验证的种植循环。

M5：文档与验证
- 更新 README 和相关 docs，标明实际已实现的种植题、仍在规划的图案内容与可选完成后花谱回看页。
- 运行单元测试和 pygame 无窗口烟雾测试。

技术建议
========
- 保留 `GameState` 服务于当前花园谜题；种植花谱更适合独立状态，因为其输入阶段不演化、验证阶段不接受播种。
- 坐标、图案、相位和迁移检测应采用集合运算，不以像素渲染结果判定规则。
- 首版只要求撤回玩家在当前尝试中新种下的花；是否支持铲除谜面已有花芽应留给后续设计确认。
- 观察/收藏页是完成图案后的补充功能，而不是首版可玩内容的替代品。

测试门槛
========
必须覆盖：
- 当前花园谜题既有测试保持通过。
- 教程测试保持通过。
- `Block` 不变、`Blinker` 周期 2、`Glider` 第 4 代平移的纯规则模型测试。
- 种植阶段移动不触发生长，播种消耗花种，撤回或重置恢复正确余额。
- 启动验证后演化不包含灯灵邻居，且输入不能中途改变布局。
- 至少一次 UI 烟雾测试：进入种植花谱，完成播种，启动生长，显示结果并重置，无异常退出。

在本机当前配置下，可优先使用：
  & 'D:\ProgramData\Anaconda3\python.exe' -m unittest discover -s tests -v

pygame 界面烟雾测试应使用 dummy SDL 驱动，避免自动弹出窗口干扰测试环境。

每次提交结果时的输出格式
========================
1. 说明本轮完成了哪个里程碑。
2. 列出具体修改文件与玩家可感知的变化。
3. 列出已执行的测试及结果。
4. 明确仍未实现的种植谜题、图案目标或札记功能，不把规划内容描述为已完成。

终止与升级条件
==============
- 如果纯规则抽取会破坏现有谜题行为，暂停 UI 扩展，先以测试修正规则边界。
- 如果种植谜题只剩被动展示而没有播种决策，不交付为完成的最小可玩版本，先收缩到一则完整可玩的纵向切片。
- 如果需要大量新增美术资源，先用现有程序绘制完成可玩原型，不把素材生成作为逻辑开发阻塞项。
- 不要未经确认把现有花园谜题转换为种植花谱；两种玩法应并存。
```

## 使用方式

该提示词是当前执行序列中的 `01`。其合理终点是：现有谜题无回归，纯规则经典行为测试通过，玩家能完成至少一则“播种后启动并验证”的种植花谱谜题。
